使用Maven构建聚合项目

​ 在最开始学习 Java 的时候我们都是吧所有的类写到一个文件中, 后来我们把类分开文件放在同一目录下, 再后来我们了解了 package 的概念, 我们对代码进行分包, 这样做都是为了方便管理代码, 可是当一个项目的代码量级很高时, 分包也很难做到方便管理, 这时, 我们就该使用分模块管理, 我们知道不同的包之间的代码可以互相依赖, 可是模块与模块之间怎样实现依赖呢?

可以使用 Maven 构建聚合工程

Maven 聚合工程: 就是可以在一个 Maven 父工程中创建多个组件(模块), 多个组件之间可以相互依赖, 实现组件复用

那么上面创建的这些个子模块都是什么意思, 都起着什么作用呢? 下面接着来看

common 模块

commom 模块旨在多个模块之间共享相同的枚举类、工具类、配置类、异常类、全局异常处理、全局返回值对象、公共依赖等等。使用 common 模块的可以尽可能的将代码的复用性提升, 但是, 若复用类的提取不符合共用的原则, 则可能导致高耦合的情况。

比如在项目的用户的客户端和后台管理端可以使用同一份公共的 common 模块以提高代码复用。

model 模块

模型工程, 所有的子工程以及所有微服务工程所涉及到的实体类都可以再此处管理

包括: POJO, PO, BO, VO, DTO, MO… , 我们来了解一下这些概念

PO(Persistant Object) 持久对象

PO是持久化对象, 用于表示数据库中的一条记录映射成的Java对象, 类中应该都是基本数据类型的包装类和String, 而不是更复杂的类型, 因为要和数据库表字段对应。PO仅仅用于表示数据, 不对数据进行操作, 拥有get和set方法。对象类中的属性对应数据库表中的字段, 有多少个字段就有多少个属性, 完全匹配。遵循JavaBean规范, 拥有get和set方法。例如:

BO(Business Object) 业务对象

BO是实际的业务对象, 会参与业务逻辑的处理操作, 里面可能会包含多个类, 用于表示一个业务对象。例如用户可以拥有宠物, 在这里把用户对应一个PO、宠物对应一个PO, 那么建立一个对应的BO对象来处理用户和宠物的关系, 每个BO都包含用户PO和宠物PO, 而处理逻辑时针对BO去处理。遵循JavaBean规范, 拥有get和set方法。例如:

但注意, BO又包括了业务逻辑, 通常在service层, 可以进行PO与VO/DTO之间的转换。

VO(Value Object) 表现对象

VO对象主要用于前端界面显示的数据, 是与前端进行交互的Java对象, 但这里是不用PO传递数据的, 因为PO包括数据库表中的所有字段, 对于前端来说我们只需要显示一部分字段就可以了, 例如我们的用户表user中的password字段、phone字段、insert_time字段是没有必要也不能显示在前端界面的。例如:

DTO(Data Transfer Object) 数据传输对象

数据传输对象是在传递给前端时使用的, 如一张表有100个字段, 那么对应的PO就有100个属性, 但是我们的前端界面只需要显示10个字段, 所以我们没必要把所有字段的PO对象传递到客户端, 我们只需要把只有这10个属性的DTO对象传递到客户端, 不会暴露服务端的表结构, 到达客户端后, 如果这个对象用于界面表示, 那么它的身份就是VO对象。

DTO和VO概念相似, 通常情况下字段也基本一致。但有所不同, DTO表示一个数据传输对象, 是在服务端用于不同服务或不同层之间的数据传输, 例如dao层到service层, service层到web层；而VO是在客户端浏览器显示的表现对象, 用于在浏览器界面数据的显示。

MO(Mongo Object) MongoDB数据库对象

和PO相似, 不过对应的是MongoDB中的字段, 可以使用Map数据类型, 在数据库中对应 JSON 键值对。

POJO(Plain Ordinary Java Object) 简单Java对象

表示一个个简单的Java对象, 而PO、VO、DTO都是典型的POJO, POJO是DO、DTO、BO、VO的统称。

dao 模块

dao(Data Access Object) 模块主要就是对数据库进行操作的, 对关系型数据库(MySQL, Oracle)和非关系型数据库(MongoDB, Redis)的操作都在这个模块下进行实现。

service 模块

service 业务模块显而易见就是用来管理 service 和 serviceImpl 的, 我们要尽量做到所有业务都在 service 中处理, controller 中只做业务调用, 这样可以降低我们代码的耦合性。

api 模块

对于一个纯后端项目或者是前后端分离项目, api 模块就是用来做接口暴露的, 这里所说的接口是http接口, 也就是我们常写的 controller , 值得一说的是 dao , service, api 这几个模块都是可以做分布式集群的(使用RPC调用), 这样可以提高项目的并发性和容灾性。

其他模块

上面介绍的这些模块也不全是必须的, 我只是介绍了这几个常见的模块, 起名规则也不是固定的, 当然可能还有其他模块, 一个项目越庞大, 拥有的模块就越多。
